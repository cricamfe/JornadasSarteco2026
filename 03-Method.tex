\section{The CASTM API}
\label{sec:openedge}

\castm{}\footnote{Available at \url{https://github.com/PhDCristian/CASTM}}, can be seen as a deterministic C-like Domain Specific Language (DSL) whose design principle is \emph{explicit spatial-temporal control with structured abstraction}: the programmer specifies what runs on which PE in which cycle, but through high-level constructs instead of raw encodings.

\subsection{API Design}
\label{subsec:lang-overview}

A \castm{} program declares a \emph{target profile}, a \emph{build configuration}, and one or more \emph{kernel} definitions. Table~\ref{tab:openedge-constructs} organizes the constructs into three layers: spatial-temporal statements (instruction placement), structured abstractions (functions, loops, labels), and architecture-aware operators for modular arithmetic idioms.

The \emph{spatial-temporal statements} layer provides direct control over instruction placement: every statement maps one-to-one to a CSV entry.
The \emph{structured abstractions} layer reduces repetition without scheduling opacity: \texttt{function} expands the function body at the call site by default (parametric macro expansion)  but can be compiled to shared subroutines under the \texttt{jump-reuse}  expansion mode, which emits each function body once and links calls through a register-based return convention.
The \emph{architecture-aware operators} layer encapsulates multi-instruction CGRA idioms for modular arithmetic, each expanding to a fixed, documented sequence.

To illustrate the contrast with manual CSV, consider broadcasting a memory load to all 16 PEs. In CSV, this requires 16 identical entries per cycle (each encoding opcode, mux selects, register indices, and immediate as 7 numeric fields): \fbox{\texttt{29,0,0,0,1,0,0}} repeated 16 times. In \castm{}, the same operation is \mbox{\texttt{bundle \{ at all: LWI R0, 0; \}}} --a single line that the compiler expands to 16 entries, eliminating an entire class of copy-paste errors--.

\subsection{Compilation Pipeline}
\label{subsec:pipeline}

The compiler performs a four-stage deterministic pipeline:
\textbf{Parse} (tokenize, syntax validate) $\to$ \textbf{Analyze} (semantic checks, function expansion, constant resolution) $\to$ \textbf{Lower} (operator expansion, loop, flat IR generation) $\to$ \textbf{Emit} (per-PE CSV encoding). Each stage is a pure transformation with no heuristic decisions. The compiler rejects ambiguous scheduling (e.g., two instructions on the same PE in the same cycle) with diagnostics reporting source location, PE coordinate, and cycle number.

\castm{} combines three properties no prior CGRA tool provides simultaneously: \emph{deterministic compilation with explicit placement} (cycle count predictable at authoring time), \emph{architecture-aware operators}
and \emph{transparent expansion} (every construct has a documented, predictable expansion to CSV).

\subsection{S-box Kernel Walkthrough}
\label{subsec:sbox-openedge}

Fig.~\ref{fig:sbox-pipeline} illustrates how our S-box kernel decomposes $x^7 \Mod p$ into four (modular) phases: 1) $x^2$ (\texttt{subrC} --squaring--); 2) $x^3 = x^2 \cdot x$ (\texttt{subrB} --general multiplication--); 3) $x^4 = (x^2)^2$ (\texttt{subrC}); and 4) $x^7 = x^4 \cdot x^3$ (\texttt{subrB}). Each phase performs a  Barrett multiplication~\cite{barrett1986} that consists of a base multiplication (\texttt{subrB}/\texttt{subrC}) followed by a Barrett reduction (\texttt{subrA}).

\input{figures/fig_barrett_structure_test}


Fig.~\ref{fig:barrett-structure} now shows how the 4-steps Barrett multiplication introduced in Algorithm~\ref{alg:barrett} is decomposed into four algorithmic stages mapped to a specific CGRA hardware capability. Step 1 represents the base product (\texttt{subrB}/\texttt{subrC}) while steps 2-4 correspond to the Barrett reduction (\texttt{subrA}). Step 1 is implemented as a parallel convolution: 32-bit operands decomposed into four 8-bit limbs, producing 16 partial products on the $4\!\times\!4$ PE mesh in our CGRA. Step 2 represents the quotient estimation and is mapped as a toroidal accumulation via nearest-neighbor routing; Step 3, the modular subtraction, avoids expensive division via subtraction and AND masking; then step 4 performs the final modular correction via branchless conditional \texttt{BSFA} (a single-cycle register select based on the ALU sign flag).

\begin{lstlisting}[caption={S-box kernel for $x^7 \Mod p$ (key excerpts). \texttt{RCR/RCL/RCT/RCB} denote \textit{Read Cell Right, Left, Top, Bottom}.}, label={lst:sbox-kernel}, breaklines=false, basicstyle=\ttfamily\fontsize{7}{7.7}\selectfont, columns=fullflexible, xleftmargin=1.5mm, numbersep=5pt, aboveskip=3pt, belowskip=1pt]
kernel "SBox" {
 // Main: set return addr, jump to subrC
 bundle { @0,1: goto subrC;
           @3,0: R3 = ZERO + ret1; }
 ret1: //(*@\textcolor{cgragreen!70!black}{$\,\triangleright$ triple merge}@*)
 bundle { @0,0: SWI R3, 720;
           @0,1: goto subrB;
           @3,0: R3 = ZERO + ret3; }
 ...
 subrC: // squaring accumulation (8 bndl)
 extract_bytes_col(R0, R1); //(*@\textcolor{cgragreen!70!black}{$\,\triangleright$ 4 PEs extract bytes}@*)
 extract_bytes_row(R0, R0);
 bundle { at all: R2 = R0 * R1; } //(*@\textcolor{cgragreen!70!black}{$\,\triangleright$ 16 products}@*)
 // Toroidal accumulation (4 bndl, RCR/RCT/RCB)
 bundle { @3,0: goto subrA; }

 subrA: // Barrett reduction (39 bndl)
 bundle { @0,0: R1 = RCR;
           @0,1: R0 = RCB;
           @0,2: R1 = RCB; }
 // Quotient estimation: 12 PEs
 bundle { at @0..2,0..3: R2 = R0 * R1; }
 // ... carry prop., normalization
 // Branchless correction:
 bundle { @0,0: R2 = R0 - RCL; }
 bundle { @0,0: BSFA R3, R0, R2, SELF;}//(*@\textcolor{cgragreen!70!black}{$\,\triangleright$ if r$\geq$p: r-p}@*)
 bundle { @3,0: goto R3; } //(*@\textcolor{cgragreen!70!black}{$\,\triangleright$ return}@*)
}

Listing~\ref{lst:sbox-kernel} shows key excerpts of the S-box kernel.
The kernel structure reveals several design decisions enabled by \castm{}:

\textbf{Jump-reuse subroutines}. All calls to subroutines are performed via register-indirect jumps. The Barrett reduction phase (\texttt{subrA}, \MetricSboxKernelSubrABundles~bundles) is shared across all four modular operations through a simple calling convention: the caller stores a return-address label in R3 (\mbox{\texttt{R3 = ZERO + retN}}, line 4, line 8), and the subroutine returns with \mbox{\texttt{goto R3}} (line 27). This avoids duplicating \MetricSboxKernelSubrABundles~bundles $\times$ \MetricSboxKernelSubrACalls~calls = \MetricSboxKernelSharedBundlesSaved~wasted bundles, reducing total kernel size from ${\sim}\MetricSboxKernelApproxPreReuseBundles$ to \MetricSboxKernelResidentBundles~bundles (see Section \ref{subsec:sbox-comparison}). In \castm{}, this pattern is expressed naturally with labels and \texttt{goto}; in manual CSV, the developer must compute absolute addresses for every jump target.

\textbf{Triple instruction merging.} At label \texttt{ret1}, three independent operations execute on three different PEs in the same cycle (lines 6-8): \texttt{SWI} stores the intermediate result, \texttt{goto} jumps to the next subroutine, and \texttt{SADD} prepares the return address. The \texttt{bundle\{\}} statement makes this spatial parallelism explicit and verifiable; in CSV, it is easy to accidentally place two instructions on the same PE, causing a silent overwrite.

\textbf{Systolic convolution.} The \texttt{subrC} uses the $4\!\times\!4$ mesh as a spatial multiplier (lines 10-14): \texttt{extract\_bytes} operators decompose operands into byte-size limbs distributed across columns and rows, then \mbox{\texttt{at all: R2 = R0 * R1}} computes all 16 partial products simultaneously. Toroidal accumulation converges results in 4~cycles via neighbor reads (RCR, RCT, RCB). All 16 PEs compute their partial product simultaneously; for squaring, the symmetric pairs are folded via in-place doubling during the subsequent accumulation phase, reducing it from 13 to 8 bundles compared to \texttt{subrB} that accumulates all the partial results explicitly.

\textbf{Branchless correction.} The final BSFA instruction (line 26) selects between $r$ and $r - p$ based on the sign flag, completing modular reduction in a single cycle without branching---avoiding control divergence across the PE mesh.

The jump-reuse feature imposes a residency constraint: the full \MetricSboxKernelResidentBundles-bundles of the kernel must remain simultaneously resident in the Configuration Register File --CRF-- so that register-indirect control flow can execute without fragmentation. When the kernel no longer fits, it must be partitioned into multiple dispatchable segments, sacrificing subroutine sharing and introducing reload overhead. Section~\ref{subsec:creg-analysis} quantifies this S-box-specific CRF-pressure trade-off and shows why it becomes a first-order architectural constraint for the full Poseidon2 evaluation.

\subsection{Spatial Mapping Methodology}
\label{subsec:spatial-mapping}

The S-box mapping above suggests a set of spatial-scheduling patterns that may be useful beyond this single kernel. Those patterns are abstracted from the concrete artifact discussed in Fig.~\ref{fig:sbox-pipeline}, Fig.~\ref{fig:barrett-structure}, and Listing~\ref{lst:sbox-kernel}. 
%However, the evidence presented in this paper is limited to the BabyBear vesion on a $4{\times}4$ OpenEdgeCGRA. Therefore, the observations presented below should be interpreted as design implications drawn from this case study, and not as universal statements applicable to other cases.



\subsubsection{Operand Decomposition across the PE Mesh}
\label{subsubsec:decomposition}

When an arithmetic operation produces intermediates wider than the PE datapath, one option is to serialize the work limb by limb on a single processing element. Our implementation instead distributes the limbs spatially across the mesh. For the 31-bit BabyBear field used here, each operand is decomposed into four 8-bit limbs aligned with rows or columns of the $4\!\times\!4$ array, exactly as summarized in Fig.~\ref{fig:barrett-structure} and instantiated through the \texttt{extract\_bytes} operators in Listing~\ref{lst:sbox-kernel}. A single multiply bundle then emits the 16 products $a_i \cdot  b_j$, and the anti-diagonal structure of those products determines how significance levels are accumulated (see Fig.~\ref{fig:mapping}(a)).

This decomposition is demonstrated directly for the BabyBear S-box studied in this paper. A related strategy should also be possible for other field widths, but the limb size, mesh dimensions, and routing budget would need to be re-tuned to the target arithmetic. For example, a 64-bit field could use wider limbs on the same mesh or more limbs on a larger array; we do not evaluate those variants here. For squaring ($a = b$), the symmetry $a_i \cdot a_j = a_j \cdot a_i$ is exploited in the accumulation schedule, which is why the current squaring path uses 8 bundles instead of the 13 bundles required by the general multiplication path.

\subsubsection{Toroidal Carry Propagation}
\label{subsubsec:carry-chain}

After partial products are formed, the current kernel moves carries and partial sums through neighbor registers rather than through a scalar loop. In \texttt{subrA} this is reflected by the \texttt{RCL/RCR/RCT/RCB} accesses highlighted in Listing~\ref{lst:sbox-kernel}. Also, normalization is expressed as a deterministic sequence of shifts, masks, additions, and transfers through the toroidal interconnect. The toroidal interconnect guaranties wrap-around connectivity, which can shorten some routes relative to a non-wrapping mesh, but the exact propagation distance still depends on placement and routing. The methodological point is therefore narrower: once limb positions are fixed, carry motion can be programmed as a repeatable spatial wavefront with architecture-visible data movement. This is illustrated in Fig.~\ref{fig:mapping}(b).



\subsubsection{Subroutine Sharing on a Stack-less Architecture}
\label{subsubsec:subroutine-sharing}

OpenEdgeCGRA does not provide a hardware call stack, so shared control flow must be constructed explicitly. In this kernel, calls are direct label jumps, the return address is stored in \texttt{R3} at a fixed PE, and the callee returns with \texttt{goto R3}, as shown in Listing~\ref{lst:sbox-kernel}. Fig.~\ref{fig:sbox-pipeline} makes clear why this matters: the same reduction body is reused across the four modular operations in the exponentiation chain. The full kernel layout also exploits contiguous placement to support asymmetric chaining: \texttt{subrC} transfers control to \texttt{subrA} through an explicit jump, whereas \texttt{subrB} reaches \texttt{subrA} by fall-through.. This convention is sufficient to share one Barrett reduction body across the four modular operations. We do not claim that the same calling discipline is optimal for every CGRA or every modular kernel; the point here is that \castm{} can express and verify it directly on the target studied in this work.



\subsubsection{Multi-PE Bundle Merging}
\label{subsubsec:merging}

A bundle activates all PEs in the same cycle, so idle PE slots are potential opportunities for control and datapath packing. The current kernel exploits this at subroutine boundaries; Listing~\ref{lst:sbox-kernel} shows the representative \texttt{ret1} case, where a store, a jump, and a return-address update occupy disjoint PEs and therefore fit into one bundle. The benefit is measurable compaction of control work into cycles that would otherwise contain idle spatial capacity; it is not correct to describe the overhead as disappearing. More generally, bundle-merging opportunities appear most naturally when control actions are localized to a small subset of PEs and can be scheduled beside arithmetic work.
